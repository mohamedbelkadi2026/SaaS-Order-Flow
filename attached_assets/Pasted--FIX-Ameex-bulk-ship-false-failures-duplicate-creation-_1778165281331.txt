Tajergrow shows: "Ameex n'a pas retourné de numéro de suivi"
But the order is actually created in Ameex's portal
I click "Réessayer" → Ameex creates a duplicate ❌
Customer gets called twice / shipping fees doubled
The platform thinks the ship failed, but it really succeeded — the response shape just didn't match what we expected.Root cause analysisThree issues stack on each other:A. Ameex returns multiple response shapesToday the code accepts only TWO shapes as success (line 854-867 in server/services/carrier-service.ts):

Shape 1: response has a tracking number → success
Shape 2: response has login: 'success' AND api.type !== 'error' → success (placeholder tracking)
But Ameex sometimes returns:

Shape 3: { status: 'ok', message: 'Parcel created' } (no login field)
Shape 4: HTTP 200 with empty body {} while the parcel was actually created
Shape 5: HTML wrapper around JSON (gateway errors)
These all fall through to "n'a pas retourné de numéro de suivi" → false failure.B. No idempotency on retryWhen the user clicks "Réessayer" on an order that:

Has NO trackNumber saved on our side (because we thought it failed)
But IS already created in Ameex
…we send a fresh creation request → Ameex creates a duplicate. There's no external_reference we send Ameex that they could use to detect "I've already seen this order".C. Webhook can't match AMEEX-PENDING-{orderNumber} placeholdersWhen the response is shape 2 (success but no tracking yet), we save AMEEX-PENDING-7211 as the trackNumber. Later, the Ameex webhook arrives with the real tracking number AMX-ABC123.
