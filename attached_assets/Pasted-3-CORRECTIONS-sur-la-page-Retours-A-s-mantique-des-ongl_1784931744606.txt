PARTIE A — Onglets "En route" et "Reçus" mal définis

PROBLÈME OBSERVÉ
- Onglet "En route" affiche À LA FOIS "En cours d'expédition au hub de
  retour" ET "Retournée" — alors que "Retournée" signifie déjà arrivé, pas
  en transit.
- Onglet "Reçus" affiche des commandes "Retournée" (statut transporteur)
  mais elles sont toutes marquées "Non confirmé" dans la colonne CONFIRMÉ —
  contradictoire : "Reçus" doit représenter les retours que L'UTILISATEUR a
  confirmés avoir physiquement reçus (via scan/bouton), pas simplement ce
  que dit le transporteur.

FIX — Redéfinir les filtres de ces 2 onglets dans le composant de la page
Retours (celui qui gère les boutons Tous/En route/Reçus/Non confirmés/
Confirmés) :

```ts
// "En route" = statuts de retour EN TRANSIT uniquement (pas encore arrivé)
const enRoute = orders.filter(o =>
  (o.status || '').toLowerCase().includes('retour') &&
  (o.status || '').toLowerCase() !== 'retournée' &&
  !o.returnConfirmedAt
);

// "Reçus" = confirmés physiquement par l'utilisateur (scan/bouton), peu
// importe le statut transporteur exact — c'est LA confirmation qui définit
// "reçu", pas le statut brut du transporteur.
const recus = orders.filter(o => !!o.returnConfirmedAt);

// "Non confirmés" = en statut retour (quel qu'il soit) mais pas encore
// confirmé physiquement — inchangé par rapport au fix précédent.
const nonConfirmes = orders.filter(o =>
  (o.status || '').toLowerCase().includes('retour') && !o.returnConfirmedAt
);

// "Confirmés" = identique à "Reçus" en pratique (les deux se basent sur
// returnConfirmedAt) — les garder comme 2 libellés d'un même filtre est OK,
// ou fusionner les onglets si ça semble redondant, au choix.
const confirmes = orders.filter(o => !!o.returnConfirmedAt);
```

Vérifier où ce filtrage est fait (client-side sur les données déjà chargées,
OU server-side via un paramètre de query) et appliquer la correction au bon
endroit — si c'est server-side, adapter la clause SQL équivalente (statut
différent de 'retournée' pour "en route", `returnConfirmedAt IS NOT NULL`
pour "reçus").

PARTIE B — Le scanner caméra ne fonctionne pas

Debugger étape par étape, car plusieurs causes possibles avec html5-qrcode :

1) Vérifier dans la console navigateur (F12) l'erreur exacte au clic sur
   "📷 Caméra" — souvent une erreur de permission, de HTTPS, ou "no cameras
   found".

2) Remplacer le composant `CameraScanner` par une version avec gestion
   d'erreur VISIBLE (pas juste `console.error`) et une liste explicite des
   caméras disponibles avant de démarrer :

```tsx
function CameraScanner({ onScan, onClose }: { onScan: (code: string) => void; onClose: () => void }) {
  const scannerRef = useRef<Html5Qrcode | null>(null);
  const elementId = "qr-reader-region";
  const [error, setError] = useState<string | null>(null);
  const [starting, setStarting] = useState(true);

  useEffect(() => {
    let scanner: Html5Qrcode;
    let cancelled = false;

    (async () => {
      try {
        // Demander explicitement la permission caméra en premier, avec un
        // message d'erreur clair si refusée/indisponible.
        const cameras = await Html5Qrcode.getCameras();
        if (cancelled) return;
        if (!cameras || cameras.length === 0) {
          setError("Aucune caméra détectée sur cet appareil.");
          setStarting(false);
          return;
        }
        scanner = new Html5Qrcode(elementId);
        scannerRef.current = scanner;
        await scanner.start(
          { facingMode: "environment" },
          { fps: 10, qrbox: { width: 250, height: 250 } },
          (decodedText) => {
            onScan(decodedText);
            scanner.stop().catch(() => {});
          },
          () => {}
        );
        setStarting(false);
      } catch (err: any) {
        console.error("[CameraScanner] failed:", err);
        setError(
          err?.name === "NotAllowedError"
            ? "Permission caméra refusée. Autorise l'accès à la caméra dans les réglages du navigateur."
            : err?.name === "NotFoundError"
            ? "Aucune caméra trouvée."
            : `Erreur caméra : ${err?.message || err}`
        );
        setStarting(false);
      }
    })();

    return () => {
      cancelled = true;
      scannerRef.current?.stop().catch(() => {});
      scannerRef.current?.clear();
    };
  }, []);

  return (
    <div className="fixed inset-0 z-50 flex flex-col items-center justify-center bg-black/80 p-4">
      {starting && <p className="text-white mb-2">Ouverture de la caméra...</p>}
      {error && <p className="text-red-400 mb-3 text-center max-w-xs">{error}</p>}
      <div id={elementId} className="w-full max-w-[320px] aspect-square bg-white rounded-lg overflow-hidden" />
      <button onClick={onClose} className="mt-4 rounded bg-white px-4 py-2 text-sm font-medium">
        Fermer
      </button>
    </div>
  );
}
```

3) Vérifier que `html5-qrcode` est bien listé dans `package.json` après le
   fix du lockfile précédent (`grep html5-qrcode package.json package-lock.json`)
   — s'il a disparu suite au nettoyage du lockfile, le réinstaller proprement :
   `npm install html5-qrcode --registry=https://registry.npmjs.org/`.

4) Tester d'abord sur desktop avec webcam (plus facile à déboguer via
   console), puis sur mobile. Si ça marche sur desktop mais pas mobile,
   c'est probablement un souci de permission navigateur mobile ou de layout
   (voir PARTIE C) qui cache l'élément vidéo.
