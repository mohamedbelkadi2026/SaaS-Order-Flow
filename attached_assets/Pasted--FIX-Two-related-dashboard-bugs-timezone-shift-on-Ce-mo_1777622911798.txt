🐛 FIX — Two related dashboard bugs: timezone shift on "Ce mois" + agent vs admin stats divergence
Bug 1: "Ce mois" includes the previous month's last day
When the user picks "Ce mois" on May 1st 2026, the filter resolves to 30/04/2026 → 01/05/2026 instead of 01/05/2026 → 01/05/2026. April 30th orders contaminate the count (e.g. 83 commandes shown when only ~5 are from May 1).
Root cause in client/src/pages/dashboard.tsx around line 65:
tsconst firstDay = new Date(now.getFullYear(), now.getMonth(), 1).toISOString().slice(0, 10);
new Date(2026, 4, 1) creates a Date at local midnight in Casablanca (UTC+1). When .toISOString() converts to UTC, it shifts back one hour → "2026-04-30T23:00:00.000Z". Slicing the first 10 chars gives "2026-04-30" instead of "2026-05-01".
This bug affects every preset that uses .toISOString() on a locally-constructed Date.
Bug 2: Same agent shows different numbers in admin view vs their own dashboard
When an admin filters by agentId=khawla + Ce mois, the cards show: Commandes 42, Confirmées 19, Refusées 2.
When khawla logs in and uses her own dashboard with the same period, the cards show: Commandes 42, Confirmées 9, Refusées 3.
Same agent, same period, different numbers → contradictory data.
Root cause: the two views hit two different endpoints:

Admin view → /api/stats/filtered (computes from getOrdersByStore filtered by assignedToId)
Agent view → /api/agents/my-stats (computes its own way, may filter differently)

The two endpoints apply slightly different status grouping. For example, "Confirmées" might mean:

/api/stats/filtered: status='confirme' OR status='expédié' OR status='delivered' OR ... (cumulative confirmed)
/api/agents/my-stats: status='confirme' only (current state only)

So the same orders are counted differently depending on which endpoint serves the page.

Fix
Step 1 — Replace ALL .toISOString().slice(0,10) calls with a timezone-safe formatter
Add this helper at the top of client/src/pages/dashboard.tsx (and any other file that does the same thing):
ts/**
 * Format a Date as YYYY-MM-DD using LOCAL calendar date.
 *
 * Avoids the .toISOString() UTC shift that turns "May 1 00:00 Casablanca"
 * into "April 30 23:00 UTC" → "2026-04-30".
 */
function ymd(d: Date): string {
  const y = d.getFullYear();
  const m = String(d.getMonth() + 1).padStart(2, '0');
  const day = String(d.getDate()).padStart(2, '0');
  return `${y}-${m}-${day}`;
}
Then replace every .toISOString().slice(0, 10) in this file with ymd(...). Specifically the date-range computation function near the top:

tsfunction computeAgentDateRange(f: { dateRange: string; dateFrom: string; dateTo: string }): { from?: string; to?: string } {
  if (f.dateRange === 'custom') {
    return { from: f.dateFrom || undefined, to: f.dateTo || undefined };
  }
  const now = new Date();
  switch (f.dateRange) {
    case 'today':     return { from: ymd(now), to: ymd(now) };
    case 'yesterday': {
      const y = new Date(now); y.setDate(y.getDate() - 1);
      return { from: ymd(y), to: ymd(y) };
    }
    case '7days': {
      const s = new Date(now); s.setDate(s.getDate() - 6);
      return { from: ymd(s), to: ymd(now) };
    }
    case 'month':
      return { from: ymd(new Date(now.getFullYear(), now.getMonth(), 1)), to: ymd(now) };
    case 'lastmonth':
      return {
        from: ymd(new Date(now.getFullYear(), now.getMonth() - 1, 1)),
        to:   ymd(new Date(now.getFullYear(), now.getMonth(), 0)),
      };
    case 'all':
    default:
      return {};
  }
}

tsfunction parseLocalYMD(s: string): Date {
  const [y, m, d] = s.split('-').map(Number);
  return new Date(y, m - 1, d);  // local midnight, no UTC shift
}
