The other dashboard cards (COMMANDES 1154, LIVRÉES 334) still correctly reflect "Mois dernier" (May), so the date range exists — it just isn't being passed into the shared `computeProfitability` function. The function is computing over its default range (current month = June, nearly empty), producing the tiny profit.

Fix so the Dashboard PROFIT NET uses the SAME period and filters as the rest of the dashboard.

## 1. Pass the dashboard's resolved range + filters into computeProfitability
In `GET /api/stats/filtered`, use the exact same `dateFrom`/`dateTo` (and product/magasin/source/city/agent filters) that the handler already computes for the other cards, and pass them through:
```ts

  dateTo,
  productId: filters.productId,
  magasinId: filters.magasinId,
  source: filters.source,
  city: filters.city,
  agentId: filters.agentId,
});
stats.profit = prof.totals.netProfit;
```
Make sure these are the resolved values (not undefined). Log them once (`console.log('[stats/filtered] profit range', dateFrom, dateTo)`) to confirm they equal the dashboard period.

## 2. computeProfitability must NOT fall back to a default when bounds are provided
Inside `computeProfitability`, only apply a default date range when `dateFrom`/`dateTo` are truly absent. When they are passed, use them verbatim. Verify it doesn't override them with "current month"/"today".

## 3. Check the unit (centimes vs DH) is consistent

## 4. Keep the "Livraison %" subtitle on the same set
