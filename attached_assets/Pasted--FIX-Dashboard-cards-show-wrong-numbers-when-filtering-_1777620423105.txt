
🐛 FIX — Dashboard cards show wrong numbers when filtering by date for agents
Problems
Bug 1: Agent-only "Refusées" card pulls from global store stats
On line 1272 of client/src/pages/dashboard.tsx:
tsx<StatCard title="Refusées" value={stats?.refused || 0} ... />
This always reads from stats (the global store stats from /api/stats/filtered), even for agents. So when an agent filters their dashboard to "Mai 2026", the Refusées card still shows the store-wide refused count from any period, mixing with the agent's other date-filtered cards.
Bug 2: Agents have two independent filter bars
Agents use agentFilters (their own filter bar at line 126), which feeds /api/agents/my-stats only. But the rest of the dashboard reads from activeFilters → /api/stats/filtered. When the agent picks "Personnalisé 01/05/2026 → 31/05/2026" in their own filter bar, it doesn't propagate to the global stats query → the cards inconsistently mix the two periods.
Bug 3: "Total Commissions à Payer" banner says "Livraisons de mai 2026" but the cards below show 40/19/8 (clearly not just May data)
The banner correctly date-filters (commissionsSummary uses activeFilters.dateFrom/dateTo), but the cards don't, because they read from /api/agents/my-stats which uses agentFilters (which defaults to "ce mois" but is set independently and may differ).
Root cause
The dashboard maintains TWO independent filter states (filters for admins, agentFilters for agents) that drive DIFFERENT endpoints (/api/stats/filtered vs /api/agents/my-stats). Cards read from a mix of both → inconsistent dates across cards.
Fix
Step 1 — Unify filter state for agents
In client/src/pages/dashboard.tsx, when the user is an agent, drive filters (the unified state) from the agent filter bar, and drop agentFilters as a separate state.
Around line 126, replace:
tsconst [agentFilters, setAgentFilters] = useState({
  dateRange: 'month' as ...,
  dateFrom: '', dateTo: '',
  city: 'all', productId: 'all',
});
With a sync hook that mirrors agent filter changes into the unified filters state:
tsconst [agentDateRange, setAgentDateRange] = useState<'today'|'yesterday'|'7days'|'month'|'lastmonth'|'all'|'custom'>('month');

// When agent picks a preset, compute the from/to and write into filters
useEffect(() => {
  if (!isAgent) return;
  const now = new Date();
  let dateFrom = '';
  let dateTo   = '';
  const fmt = (d: Date) => d.toISOString().slice(0, 10);

  if (agentDateRange === 'today') {
    dateFrom = dateTo = fmt(now);
  } else if (agentDateRange === 'yesterday') {
    const y = new Date(now); y.setDate(y.getDate() - 1);
    dateFrom = dateTo = fmt(y);
  } else if (agentDateRange === '7days') {
    const f = new Date(now); f.setDate(f.getDate() - 6);
    dateFrom = fmt(f); dateTo = fmt(now);
  } else if (agentDateRange === 'month') {
    dateFrom = fmt(new Date(now.getFullYear(), now.getMonth(), 1));
    dateTo   = fmt(now);
  } else if (agentDateRange === 'lastmonth') {
    dateFrom = fmt(new Date(now.getFullYear(), now.getMonth() - 1, 1));
    dateTo   = fmt(new Date(now.getFullYear(), now.getMonth(), 0));
  } else if (agentDateRange === 'all') {
    dateFrom = ''; dateTo = '';
  }
  // 'custom' falls through — uses the dates the user explicitly picked

  if (agentDateRange !== 'custom') {
    setFilters(prev => ({ ...prev, dateFrom, dateTo }));
  }
}, [agentDateRange, isAgent]);
For agents, also force agentId = current_user_id in filters:
tsuseEffect(() => {
  if (isAgent && user?.id) {
    setFilters(prev => ({ ...prev, agentId: String(user.id) }));
  }
}, [isAgent, user?.id]);
This means when an agent is logged in:

Their filter bar drives the SAME filters state that admins use
Server gets agentId=<them> in every query → all stats pre-scoped to them
A single dateFrom/dateTo flows to every endpoint
