🐛 FIX — Manual totalPrice override is lost when saving an existing order
My problem
When I edit an existing order in the order details modal:

Order has 3 items × 799 DH = 2397 DH total
I want to apply a special offer: customer pays only 999 DH for the bundle
I type 999 in the "Prix total" field
I click "Enregistrer"
Modal closes, saves successfully
I reopen the same order → "Prix total" is back to 2397 DH ❌

The override I typed is silently overwritten. I want my manual value to be persisted exactly as typed and preserved across reopens.
Investigation done
I already verified:

The backend /api/orders/:id PATCH endpoint accepts totalPrice and writes it as-is (no recomputation in server/routes.ts line 4137 or in storage.updateOrderItem)
The save payload at client/src/components/order-details-modal.tsx line 332 sends totalPrice correctly: Math.round(parseFloat(fields.totalPrice || "0") * 100)
No useEffect in the modal recomputes fields.totalPrice based on items

So either:

(a) Some code path I haven't found is silently recomputing totalPrice in the save flow
(b) The PATCH succeeds but the row read back uses a different totalPrice (cached items × price)
(c) The Replit Agent introduced a useEffect we missed

Step 4 — Make updateOrder in storage trust the totalPrice exactly
Find the actual update path. If storage.updateOrder exists, ensure it doesn't override totalPrice:
ts// server/storage.ts — updateOrder method
async updateOrder(id: number, data: Partial<Order>): Promise<Order | undefined> {
  // CRITICAL: Do NOT recompute totalPrice from items. The caller's value is

Step 5 — Save totalPrice BEFORE syncing items
In the modal save mutation (line 335-358), the order is patched first, then items are synced. That's correct order. But verify item PATCH doesn't trigger a server-side recompute. If it does, swap the order: sync items first, then PATCH the order with the manual totalPrice as the LAST write. That way nothing overwrites it after.
ts// 1. Sync all items first
for (const item of localItems) {
  // ... existing item POST/PATCH ...
}

// 2. THEN patch the order with the explicit totalPrice — last writer wins
const res = await apiRequest("PATCH", `/api/orders/${order.id}`, payload);
const updatedOrder = await res.json();

return updatedOrder;
This makes the user's totalPrice the last write, immune to any side effect of item updates.
