🐛 FIX (final) — Modal reopens with stale totalPrice; force fresh fetch + add traces
Status
The previous fix succeeded half-way:

✅ Save persists totalPrice = 2200 to the DB (the orders list shows 2200 correctly)
❌ Reopening the order modal shows 2397.00 DH — the items × price computed value, NOT the saved 2200

So the bug is in the load path of the modal, not the save path. The modal is reading a stale order prop (from the parent's filtered-orders cache) where totalPrice is still the old computed value.
Fix
Step 1 — Modal: fetch fresh order data on open, don't trust the prop
In client/src/components/order-details-modal.tsx, around the top of the component (after the useState declarations near line 235), add a query that always refetches the single order from /api/orders/:id when the modal opens. Use that fresh value instead of the prop.
tsximport { useQuery } from '@tanstack/react-query';

// ... existing state ...

// Always fetch the latest order from the server when the modal is open.
// The prop may be stale (cached from a list query that ran before the last save).
const { data: freshOrder } = useQuery({
  queryKey: ['/api/orders', order?.id],
  enabled: !!order?.id,
  queryFn: async () => {
    const res = await fetch(`/api/orders/${order!.id}`, { credentials: 'include' });
    if (!res.ok) throw new Error('Failed to fetch order');
    return res.json();
  },
  staleTime: 0,            // always refetch on open
  refetchOnMount: true,
});

// Use freshOrder when available, fall back to the prop on first render
const effectiveOrder = freshOrder ?? order;
Then change the useEffect at line 265 to depend on effectiveOrder instead of order:

The console log will tell us in the browser console:

fromProp_totalPrice = what the parent passed (could be stale)
fromFresh_totalPrice = what the server actually has now
finalUsedDH = what the modal ended up showing
itemsSum = what 799 × 3 = 2397 cents

So if the bug repeats:

fromFresh = 220000, finalUsed = 2200.00 → fixed ✅
fromFresh = 239700 → server actually has 2397 in DB → real backend bug, look elsewhere
fromFresh = 220000, finalUsed = 2397.00 → modal logic still broken, need to recheck the useEffect

Step 2 — Invalidate the single-order cache key on save too
In onSuccess of the save mutation (line 363-367), add the single-order cache key:
tsonSuccess: (updatedOrder: any) => {
  toast({ title: "Commande mise à jour avec succès" });
  queryClient.invalidateQueries({ queryKey: ["/api/orders"] });
  queryClient.invalidateQueries({ queryKey: ["/api/orders/filtered"] });
  queryClient.invalidateQueries({ queryKey: ["/api/all-orders"] });
  queryClient.invalidateQueries({ queryKey: ["/api/stats"] });
  queryClient.invalidateQueries({ queryKey: ["/api/products"] });

  // NEW: refresh the single-order query so the modal re-renders with fresh data immediately
  if (order?.id) {
    queryClient.invalidateQueries({ queryKey: ['/api/orders', order.id] });
  }

  const saved = updatedOrder ?? { ...order, ...fields };
  onUpdated?.(saved);
  const statusChanged = (updatedOrder?.status ?? fields.status) !== order?.status;
  if (statusChanged) onClose();
},
Without this, even after save, the useQuery(['/api/orders', orderId]) keeps the previously-loaded result and doesn't refetch.
Step 3 — Server-side: log on GET so we can verify the DB value
In server/routes.ts line 1520:
tsapp.get("/api/orders/:id", requireAuth, async (req, res) => {
  const orderId = Number(req.params.id);
  const order = await storage.getOrder(orderId);
  if (!order) return res.status(404).json({ message: "Order not found" });
  if (order.storeId !== req.user!.storeId && !req.user!.isSuperAdmin) {
    return res.status(403).json({ message: "Accès refusé" });
  }
  if (req.user!.role === 'agent' && order.assignedToId !== req.user!.id) {
    return res.status(403).json({ message: "Accès refusé" });
  }

  console.log(`[GET /api/orders/${orderId}] returning totalPrice=${order.totalPrice} (${(order.totalPrice / 100).toFixed(2)} DH), itemsCount=${(order as any).items?.length ?? 0}`);
  res.json(order);
});
This way, you'll see in Railway logs every time the modal opens whether the DB really has 220000 or 239700.
