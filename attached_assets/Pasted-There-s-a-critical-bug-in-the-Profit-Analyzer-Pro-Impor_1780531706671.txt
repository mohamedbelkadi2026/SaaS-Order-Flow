ROOT CAUSE HYPOTHESES
═══════════════════════════════════════════════════════════════════════
1. The Aperçu table reads from a different state/array than the Sélection section.
2. Two different functions compute quantity:
   - One in detectProducts() (used by Sélection) — works correctly
   - Another inline in the Aperçu render — broken

4. The QUANTITÉ mapping dropdown was set to "Ref" (visible in screenshot), which is wrong — the user mapped "Ref" as both PRODUIT and QUANTITÉ. The Aperçu may be using the QUANTITÉ mapping naively, while Sélection uses row counts.

STEP 2 — FIX
═══════════════════════════════════════════════════════════════════════
The fix has two parts:

A) Unify the quantity computation. Both Sélection AND Aperçu must use the SAME source of truth. The Sélection section is correct, so make the Aperçu use the same computation.

Create a single helper function (place it where the parsing logic lives):

  function computeProductStats(rows, mapping) {
    // rows = parsed CSV rows (delivered only, after status filter)
    // mapping = { product, quantity, price, status, shipping }
    const stats = new Map();

    for (const row of rows) {
      const productKey = String(row[mapping.product] || "").trim();
      if (!productKey) continue;

      // QUANTITY: prefer the mapped quantity column.
      // If quantityColumn is set and parses to a positive number, use it.
      // Otherwise default to 1 per row (one row = one delivery of 1 unit).
      let qty = 1;
      if (mapping.quantity && mapping.quantity !== "(aucune)") {
        const raw = row[mapping.quantity];
        const parsed = parseFloat(String(raw).replace(",", "."));
        if (!isNaN(parsed) && parsed > 0) qty = parsed;
      }

      const price = parseFloat(String(row[mapping.price] || 0).replace(",", ".")) || 0;
      const shipping = parseFloat(String(row[mapping.shipping] || 0).replace(",", ".")) || 0;

      if (!stats.has(productKey)) {
        stats.set(productKey, {
          productKey,
          productName: productKey,
          livraisons: 0,
          totalQty: 0,
          totalRevenue: 0,
          totalShipping: 0,
        });
      }
      const s = stats.get(productKey);
      s.livraisons += 1;        // count of rows = count of deliveries
      s.totalQty += qty;        // sum of quantities
      s.totalRevenue += price;
      s.totalShipping += shipping;
    }

    return Array.from(stats.values());
  }

Use this function for BOTH:
- The Sélection section (each card shows `livraisons` and `totalRevenue`)
- The Aperçu produits table (QTÉ shows `totalQty`, CA BRUT shows `totalRevenue`, FRAIS LIVR shows `totalShipping`)

KEY DECISION:
- If the column mapped to QUANTITÉ is "Ref" (a non-numeric ID) — which is exactly what happened in the screenshot — `parseFloat` returns NaN, so we fall back to qty=1 per row. That means QTÉ = number of rows = number of livraisons (matches Sélection).
- If user maps a real numeric Quantity column, qty per row uses that value.

B) Validate quantity values. If parsed qty is negative OR over a sane threshold (e.g. > 10000 per row), treat it as invalid and fall back to 1. Negative and huge values are clearly parsing errors:

  if (!isNaN(parsed) && parsed > 0 && parsed <= 10000) qty = parsed;

C) Guard the mapping. In the column mapping dropdowns, prevent the user from mapping the same column to both PRODUIT and QUANTITÉ. If they pick the same column for both, ignore the QUANTITÉ mapping (treat as "(aucune)") and show a warning:

  {mapping.product && mapping.quantity === mapping.product && (
    <div className="text-xs text-amber-400 mt-1">
      ⚠️ La même colonne ne peut pas servir pour Produit et Quantité. Quantité ignorée (1 unité par livraison).
    </div>
  )}
