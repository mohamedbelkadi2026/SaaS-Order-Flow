🐛 FIX — Ameex rejection "Destinataire: est obligatoire" — validate fields before shippingMy problemBulk-ship to Ameex returns:
json{ "login": "success", "api": { "type": "error", "msg": "Destinataire: est obligatoire" } }Translated: "Recipient name is required". Ameex is correctly rejecting orders that have an empty/null customerName. But my orders DO have customer names visible in the UI — so either:

The customer name has invisible characters (zero-width spaces, RTL marks, BOM)
The customer name is null in the database for that specific order
There's some character that fails Ameex's server-side validation
Today the platform sends the request anyway and surfaces a confusing error. I want:

Pre-flight validation before sending — refuse to ship if mandatory fields are missing
Field cleaning — strip invisible characters, fix RTL, normalize whitespace
Clear error messages when a field is genuinely missing
Fix — Ameex-only, no other carrier touchedStep 1 — Add a pre-flight validator for Ameex shipmentsIn server/services/carrier-service.ts, just above buildAmeexPayload (line 365), add:ts/**
 * Validate the Ameex-mandatory fields BEFORE making the API call.
 * Returns a list of missing/invalid fields, or empty array if OK.
 *
 * Ameex requires: destinataire (customer name), telephone, ville, adresse,
 * montant, produit. Missing any of these = rejection.
 */
function validateAmeexInput(input: CarrierShipInput): string[] {
  const issues: string[] = [];

  // Strip invisible chars (RTL marks, zero-width spaces, BOM) before checking
  const cleaned = (s: any) => String(s ?? '')
    .replace(/[\u200B-\u200F\u202A-\u202E\u2066-\u2069\uFEFF]/g, '')  // invisible
    .replace(/\s+/g, ' ')                                              // collapse whitespace
    .trim();

  const name    = cleaned(input.customerName);
  const phone   = cleaned(input.phone);
  const city    = cleaned(input.city);
  const address = cleaned(input.address);
  const product = cleaned(input.productName);

  if (!name || name.length < 2)       issues.push('Nom du destinataire manquant ou trop court');
  if (!phone || phone.length < 8)     issues.push('Téléphone manquant ou invalide');
  if (!city)                          issues.push('Ville manquante');
  if (!address)                       issues.push('Adresse manquante');
  if (!product)                       issues.push('Nom du produit manquant');
  if (!input.totalPrice || input.totalPrice <= 0) issues.push('Prix total manquant ou nul');

  return issues;
}The cleaning regex strips:

\u200B-\u200F — zero-width space, ZWNJ, ZWJ, LRM, RLM
\u202A-\u202E — directional formatting (LRE, RLE, PDF, LRO, RLO)
\u2066-\u2069 — directional isolates
\uFEFF — byte order mark
These often hide in customer names imported from Shopify/WooCommerce (RTL Arabic store fields), and they look "present" in the UI but get stripped server-side or fail validation on third-party APIs.Step 2 — Use cleaned values in the actual payloadUpdate buildAmeexPayload (line 365) to clean the same way, so even if validation passes, what we send is normalized:tsfunction buildAmeexPayload(input: CarrierShipInput): Record<string, unknown> {
