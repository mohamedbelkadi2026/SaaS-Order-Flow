Change 3 — Auth header fallback
Looking at the Ameex credentials UI, the customer fills in only one credential: Authorization. Currently the code uses two headers:
ts'C-Api-Key': cleanKey(apiKey),
'C-Api-Id':  cleanKey(apiSecret || ''),
If only apiKey is set (the Authorization value), C-Api-Id is empty → Ameex rejects.
Two possible behaviors Ameex supports:

A: Use Authorization: Bearer <token> (single header)
B: Use C-Api-Key + C-Api-Id (split header)

Add a fallback that sets Authorization too, since the credential UI explicitly calls it Authorization:
In the Ameex POST section, find the headers: block (around line 829) and replace:

  'C-Api-Id':  cleanKey(apiSecret || ''),
},
With:
tsheaders: {
  ...fd.getHeaders(),
  // Send under multiple auth header names — Ameex picks the one it expects.
  // Different Ameex API versions / portal configs use different names.
  'C-Api-Key':     cleanKey(apiKey),
  'C-Api-Id':      cleanKey(input.apiSecret || apiKey || ''),
  'Authorization': `Bearer ${cleanKey(apiKey)}`,
  'Token':         cleanKey(apiKey),
  'X-Api-Key':     cleanKey(apiKey),
},
The same value sent under multiple header names — whichever Ameex's gateway uses will match. Extra headers are ignored.
