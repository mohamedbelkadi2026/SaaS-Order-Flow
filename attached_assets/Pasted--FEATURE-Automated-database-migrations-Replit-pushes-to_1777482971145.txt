
FEATURE — Automated database migrations: Replit pushes to GitHub → Railway auto-deploys → migration runs automatically against Railway Postgres
My setup

Replit = code editor only. I push code to GitHub from here.
GitHub = source of truth.
Railway = production hosting. It pulls from GitHub on every push and redeploys. The Postgres database lives on Railway.
tajergrow.com = my domain pointing to the Railway service.

My problem
For weeks, I've been pasting SQL migration blocks here, and Replit Agent has been running them against the Replit-local Postgres (which is unused). My production database on Railway never received any of those ALTER TABLE statements. Result: every multi-magasin feature in the codebase is silently broken because the columns don't exist in the live DB.
What I want
A migration system that runs automatically on Railway every time I deploy. Specifically:

A migrations/ folder at the repo root containing every pending ALTER TABLE / CREATE INDEX / UPDATE as a numbered .sql file.
A migration runner script that, on Railway boot, scans the folder and applies any migration that hasn't been applied yet (tracked in a _migrations table).
The Railway start command runs the migrator BEFORE the web server starts.
Idempotent — re-running the same migration is safe (uses IF NOT EXISTS, etc.).
Logs every migration as it runs so I can see in Railway logs exactly what was applied.
Backfills the existing missing schema in one initial migration so the production DB catches up to the codebase immediately.

2. server/migrate.ts — NEW migration runner
tsimport { Pool } from 'pg';
import { readdirSync, readFileSync } from 'fs';
import { join, basename } from 'path';

/**
 * Lightweight migration runner.
 * - Reads every .sql file in /migrations sorted by filename
 * - Applies any not yet recorded in the _migrations table
 * - Each migration runs in its own transaction
 * - Idempotent: rerunning a migration that's already applied is a no-op
 *
 * Naming convention: NNNN_description.sql (e.g. 0001_add_magasin_id.sql)
 */
export async function runMigrations(connectionString: string): Promise<void> {
  const pool = new Pool({
    connectionString,
    // Railway requires SSL for external connections; for in-network DB, this also works
    ssl: connectionString.includes('railway') || connectionString.includes('rlwy.net')
      ? { rejectUnauthorized: false }
      : false,
  });

  try {
    // Bootstrap the tracking table
    await pool.query(`
      CREATE TABLE IF NOT EXISTS public._migrations (
        id          serial PRIMARY KEY,
        filename    text   NOT NULL UNIQUE,
        applied_at  timestamp NOT NULL DEFAULT now(),
        checksum    text
      );
    `);

    const migrationsDir = join(process.cwd(), 'migrations');
    let files: string[];
    try {
      files = readdirSync(migrationsDir)
        .filter(f => f.endsWith('.sql'))
        .sort(); // lexicographic = numeric prefix order
    } catch (e: any) {
      if (e.code === 'ENOENT') {
        console.log('[MIGRATE] No /migrations folder — skipping');
        return;
      }
      throw e;
    }

    if (files.length === 0) {
      console.log('[MIGRATE] No migration files to apply');
      return;
    }

    const { rows: applied } = await pool.query<{ filename: string }>(
      'SELECT filename FROM public._migrations'
    );
    const appliedSet = new Set(applied.map(r => r.filename));

    let count = 0;
    for (const file of files) {
      if (appliedSet.has(file)) {
        console.log(`[MIGRATE] ✓ ${file} (already applied)`);
        continue;
      }

      const sql = readFileSync(join(migrationsDir, file), 'utf-8');
      console.log(`[MIGRATE] → Applying ${file} (${sql.length} bytes)…`);

      const client = await pool.connect();
      try {
        await client.query('BEGIN');
        await client.query(sql);
        await client.query(
          'INSERT INTO public._migrations (filename) VALUES ($1)',
          [file]
        );
        await client.query('COMMIT');
        console.log(`[MIGRATE] ✓ ${file} applied successfully`);
        count++;
      } catch (err: any) {
        await client.query('ROLLBACK').catch(() => {});
        console.error(`[MIGRATE] ✗ ${file} FAILED — rolled back`);
        console.error(`[MIGRATE]   ${err.message}`);
        throw err; // halt boot — do NOT start the server with a broken schema
      } finally {
        client.release();
      }
    }

    console.log(`[MIGRATE] Done. ${count} new migration(s) applied.`);
  } finally {
    await pool.end();
  }
}
3. server/index.ts — call the runner BEFORE starting the server
Find the top of server/index.ts (or wherever the express app boots). Add right at the start, before app.listen or any other initialization:
tsimport { runMigrations } from './migrate';

const DATABASE_URL = process.env.DATABASE_URL;
if (!DATABASE_URL) {
  console.error('[BOOT] DATABASE_URL is not set — refusing to start');
  process.exit(1);
}

// ── Run pending migrations BEFORE accepting traffic ─────────────────────
// This guarantees that by the time the web server is up, the schema matches
// what the codebase expects. If a migration fails, the server does not start —
// Railway will keep the previous deploy serving traffic.
(async () => {
  try {
    console.log('[BOOT] Checking for pending migrations…');
    await runMigrations(DATABASE_URL);
  } catch (err: any) {
    console.error('[BOOT] Migration failed — aborting boot. Previous deploy stays live.');
    console.error(err);
    process.exit(1);
  }

  // Now bring up the rest: express app, websockets, background sync, etc.
  // (whatever your existing boot code already does)
  // ...
})();
If your boot code is already wrapped in a function (bootstrap(), start(), etc.), call runMigrations as the first line of that function and await it.
4. package.json — make sure pg is installed
If it isn't already (it should be — Drizzle uses it), add it:

5. Railway start command — should already work
Railway's default start command is npm start, which runs your existing build entry. As long as server/index.ts is the entry, the migrate call runs on every boot. No Railway config change needed.
If your Railway has a custom start command (Settings → Service → "Start Command"), make sure it points to the same entry.
