🎯 ROOT CAUSE — Lead percentages are stored GLOBALLY per agent, not per-magasin
I read the code carefully and found the actual structural bug. The previous fixes correctly moved distributionMethod, agentIds, and distributionEpoch to the per-magasin level. But leadPercentage was never moved. It still lives on store_agent_settings.storeId which is the parent account (admin.storeId), not the magasin.
Proof in the code
server/routes.ts line 4419:
tsawait storage.upsertStoreAgentSetting(userId, admin.storeId!, settingsPayload);
//                                            ^^^^^^^^^^^^^^^
//                                            parent account ID, NOT magasinId
server/storage.ts line 1849 (inside getNextAgent):
tsconst settings = await db.select().from(storeAgentSettings)
  .where(eq(storeAgentSettings.storeId, storeId));   // ← reads parent, ignores magasin
So even if the user sets:

Magasin 1: khawla 50%, naima 25%, fatima 25%
Magasin 2: khawla 100%

…the database physically can only store ONE row per agent (because of how store_agent_settings is keyed). Whichever value was saved last overwrites the other. There's effectively only one global percentage per agent across the whole account.
Symptom (matches my screenshot exactly)
Total assigned = 66, distributed 26/21/19 ≈ 39%/32%/29%. That looks suspiciously close to round-robin (33/33/33), NOT 50/25/25. This happens because:

The single global percentage row IS being read (so getNextAgent enters the pourcentage branch).
But the distributionEpoch reset clears lastAssignedAgentId, the count window resets to fresh, and with all agents at 0 count → all three have the same maximum deficit on the first order. Tiebreaker is array order, so the first agent in eligibleAgents keeps winning until the deficit shifts.
With three agents all at 50/25/25 (or whatever was stored last), the deficit math eventually equalizes to roughly even distribution — NOT the configured ratios — when there's only one global setting fighting against multi-magasin counts being commingled.

THE FIX — Make store_agent_settings per-magasin
This is the correct architectural fix. After this, an agent can have different roles + different percentages in different magasins, which is what a multi-tenant SaaS needs.
Step 1 — Schema: add magasin_id to store_agent_settings
In shared/schema.ts, find the storeAgentSettings table definition and add a magasinId column:

Step 3 — upsertStoreAgentSetting becomes per-magasin
In server/storage.ts (line ~2075), update the signature and the upsert key:
tsasync upsertStoreAgentSetting(
  agentId: number,
  storeId: number,
  magasinId: number | null,         // ← NEW
  data: { roleInStore?: string; leadPercentage?: number; allowedProductIds?: string; allowedRegions?: string; commissionRate?: number }
): Promise<StoreAgentSetting> {
  const existing = await db.select().from(storeAgentSettings)
    .where(and(
      eq(storeAgentSettings.agentId, agentId),
      eq(storeAgentSettings.storeId, storeId),
      magasinId == null
        ? isNull(storeAgentSettings.magasinId)
        : eq(storeAgentSettings.magasinId, magasinId),
    ))
    .limit(1);

  if (existing[0]) {
    const [updated] = await db.update(storeAgentSettings)
      .set({ ...data })
      .where(eq(storeAgentSettings.id, existing[0].id))
      .returning();
    return updated;
  }
  const [created] = await db.insert(storeAgentSettings)
    .values({ agentId, storeId, magasinId, ...data })
    .returning();
  return created;
}
Update the interface declaration on line ~135 to match the new signature.
Step 4 — getNextAgent reads per-magasin settings, with graceful fallback
In server/storage.ts line 1848, replace the settings load with a magasin-scoped + fallback read:
ts// 3. Load per-magasin agent settings, falling back to legacy account-level (magasin_id IS NULL)
const settingsConditions: any[] = [eq(storeAgentSettings.storeId, storeId)];
const settings = await db.select().from(storeAgentSettings).where(and(...settingsConditions));

// Build a per-agent map: prefer magasin-scoped row, fall back to legacy NULL row
const settingsMap = new Map<number, typeof settings[number]>();
for (const s of settings) {
  const isMatch = magasinId ? s.magasinId === magasinId : s.magasinId == null;
  const isLegacy = s.magasinId == null;
  const existing = settingsMap.get(s.agentId);
  if (isMatch || (!existing && isLegacy)) {
    settingsMap.set(s.agentId, s);
  }
}
This resolves: "if a magasin-specific setting exists, use it; otherwise fall back to the global default row (magasin_id IS NULL)."

ACCEPTANCE CRITERIA

✅ \d store_agent_settings shows magasin_id column.
✅ Team page agent-edit modal shows ONE percentage row per linked magasin (not a single global field).
✅ Setting khawla 50% in Magasin 1 and 100% in Magasin 2 saves both rows in store_agent_settings.
✅ Order arriving on Magasin 1's webhook → distribution respects Magasin 1's percentages, not Magasin 2's.
✅ Server logs show [DIST-%] agent=X target=50.0% ... matching the per-magasin config.
✅ Resetting Magasin 1's epoch does NOT affect Magasin 2's count window.
✅ /api/_debug/distribution/<magasinId> returns the right percentages for that magasin.
✅ No TypeScript errors. Existing legacy data (settings with magasin_id IS NULL) still works as account-wide fallback.
