🐛 FIX — Bulk shipping creates duplicates + only 1 order moves to "Suivi des Colis"
Two bugs in one operation
Bug 1: Already-shipped orders get re-shipped (DUPLICATE TRACKING NUMBERS)
In server/storage.ts line 632-641, bulkShipOrders includes orders already in status 'expédié' and 'Attente De Ramassage' — meaning if a user clicks "Sélectionner tout" and the list contains orders already dispatched, the code re-sends them to Digylog. Carrier creates new tracking numbers → 2-3-4 duplicates per order in their system.
Bug 2: Status update race condition
In server/routes.ts line 1401-1404, after a successful carrier API call, the code does:
tsdbUpdates.push(
  storage.updateOrderShipping(order.id, trackingNumber, labelUrl!, provider),
  storage.updateOrderStatus(order.id, 'Attente De Ramassage'),
);
These two writes race each other because they're pushed into dbUpdates[] and executed in parallel later. Sometimes updateOrderShipping overwrites status, sometimes updateOrderStatus runs first then updateOrderShipping clobbers it back to 'confirme'. Result: only 1-2 orders out of 10 actually move statuses.
The fix
Step 1 — server/storage.ts line 632-641: ONLY 'confirme' is shippable, AND skip orders that already have a track number
Replace bulkShipOrders with:
tsasync bulkShipOrders(orderIds: number[], storeId: number): Promise<Order[]> {
  if (orderIds.length === 0) return [];

  const eligible = await db.select().from(orders)
    .where(and(
      inArray(orders.id, orderIds),
      eq(orders.storeId, storeId),
      // Only confirmed orders are shippable.
      // Do NOT include 'expédié' or 'Attente De Ramassage' — those are
      // already-shipped states and re-sending creates duplicate tracking
      // numbers in the carrier's system.
      eq(orders.status, 'confirme'),
      // Defense-in-depth: even if status was wrongly left as 'confirme',
      // a non-null track_number means the order was already dispatched.
      sql`(${orders.trackNumber} IS NULL OR ${orders.trackNumber} = '')`,
    ));

  return eligible;
}
The change in plain language:

Old: status ∈ {confirme, expédié, Attente De Ramassage} → eligible
New: status = 'confirme' AND track_number IS empty → eligible

This kills the duplicate-shipping bug at its source. Even if the user clicks "Expédier" 10 times in a row on the same selection, after the first successful pass every order has a track_number and is filtered out on subsequent passes.
Step 2 — server/routes.ts line ~1399-1410: serialize the post-shipping DB writes per order
The current code pushes both writes into dbUpdates[] and runs them via Promise.all later. That's the race. Replace the success branch (around line 1399-1414):
ts} else {

Important: storage.updateOrder already exists and accepts a partial Order. We're using ONE call that sets all five fields (trackNumber, labelUrl, shippingProvider, status, lastActionAt) atomically. No more split between updateOrderShipping + updateOrderStatus.
Don't delete updateOrderShipping and updateOrderStatus — other parts of the codebase still call them. Just stop calling them in this specific bulk-ship loop.
